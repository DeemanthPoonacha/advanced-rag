Acme Corp Systems - Server Troubleshooting & Operations Guide
=============================================================

This document provides step-by-step procedures for addressing common server alerts, database connectivity failures, and container orchestration issues.

Issue 1: Redis Cache Connection Timeout (Error Code: ERR_REDIS_TIMEOUT)
------------------------------------------------------------------------
Symptoms:
- UI displays "Cache unavailable" or shows high latency.
- Backend logs show: "ConnectionTimeoutError: Redis connection lost".

Steps to resolve:
1. SSH into the cache node:
   ssh admin@cache-node-01.internal.acme.com
2. Check the Redis service status:
   sudo systemctl status redis-server
3. If the service is stopped, restart it:
   sudo systemctl restart redis-server
4. Verify memory usage:
   redis-cli info memory
   If memory usage is above 95%, flush the volatile keys:
   redis-cli FLUSHALL ASYNC

Issue 2: PostgreSQL Database Connection Pool Exhausted (Error Code: ERR_DB_POOL_FULL)
-------------------------------------------------------------------------------------
Symptoms:
- APIs return HTTP 500 Internal Server Error.
- Log message: "FATAL: remaining connection slots are reserved for non-replication superuser connections".

Steps to resolve:
1. Run the active query inspector script to identify long-running or leaked connections:
   python3 /opt/acme/scripts/inspect_db_conns.py --threshold-seconds 60
2. Terminate rogue backends using the PostgreSQL kill command:
   SELECT pg_terminate_backend(pid) FROM pg_stat_activity WHERE state = 'active' AND now() - query_start > interval '5 minutes';
3. Adjust pool size in configuration file `/etc/acme/api.conf` by setting:
   MAX_CONNECTION_POOL_SIZE=50
4. Reload the API service:
   sudo systemctl reload acme-api-service

Issue 3: Kubernetes Pod Out of Memory (OOMKilled)
-------------------------------------------------
Symptoms:
- Pod status changes to "CrashLoopBackOff".
- kubectl describe pod shows: "Last State: Terminated, Reason: OOMKilled, Exit Code: 137".

Steps to resolve:
1. Identify the offending pod and namespace:
   kubectl get pods -A --field-selector status.phase=Failed
2. Check the memory usage limits in the deployment spec:
   kubectl get deployment api-gateway -o yaml | grep -A 5 resources
3. Increase the memory limit by editing the deployment:
   kubectl edit deployment api-gateway -n production
   Change the resources limits memory from "512Mi" to "1Gi".
4. Monitor the rollout of the new pod:
   kubectl rollout status deployment/api-gateway -n production
